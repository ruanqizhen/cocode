# 工程级迭代与持续验证

> **“软件工程没有银弹，但‘小步迭代、及时反馈’是人机协同开发中最接近银弹的真理。”**

---

## “一次只做一件事”：降低单次生成规模的克制之道

面对大模型的超凡速度，开发者最容易犯的错误就是**胃口太大**：一次性命令 AI 修改 5 个不相关的核心文件，或是把重写整个模块的重任一股脑丢过去。

这种粗放型的生成方式会导致灾难性的 Bug 爆发，最后让你陷入漫长而绝望的手工调试泥潭。

:::important 克制之道
在人机协作中，越克制，反而越快速。
:::

### 🎯 “微操作（Micro-operations）”原则
* 每次交互只解决一个单一、明确的任务点。
* 例如：不要说“帮我实现完整的结账支付和库存扣减”。
* 改为拆解为四个极小步骤：
  1. *第一步*：仅修改数据库，加入支付状态字段。
  2. *第二步*：编写计算应付金额的纯函数，并补齐边界测试。
  3. *第三步*：接入 Stripe 支付网关 SDK，仅实现调用与回调。
  4. *第四步*：编写库存扣减事务，与支付回调进行集成。

每次生成保持在 50 行以内的核心变动，人类审查的难度极低，系统能以最稳健的姿态滚滚向前。

## AI 驱动的测试驱动开发（TDD）：让 AI 写测试，再让 AI 跑通测试

测试驱动开发（TDD）在传统的开发工作中被认为“过于繁琐”，因为写测试用例是一件枯燥、重复的工作。但在 AI 时代，TDD 迎来了它的黄金涅槃。

### 🌟 人机同频下的 TDD 闭环
1. **人类定契约**：在 Spec 中写明输入和输出的严格边界。
2. **AI 写测试用例**：命令 AI 根据契约，编写覆盖“正常路径、边界零值、异常抛错”的全套单元测试（Unit Tests）。
3. **人类运行并确认红灯**：运行测试，毫无疑问，测试由于主逻辑未编写而爆红。
4. **AI 编写主代码**：把测试代码贴给 AI，命令它：“编写业务类，直到完全通过上面的测试，不许漏掉任何一个 Case。”
5. **完全跑通（绿灯）**：代码瞬间通过测试，架构坚固如铁。

## 自动化卡点验证：在每次 Tab 键后运行类型检查与 Lint 工具

在人机协作中，哪怕是 GitHub Copilot 在你按下 Tab 键后生成的区区几行代码，都可能暗藏着拼写错误、类型缺失或语法漏洞。

### 构建你的“赛博安全带”
不要等到整个功能写完才去编译。你必须建立起极高频、甚至自动化的**卡点验证体系**：

```bash
# 每一个小改动后，在终端立即运行卡点命令
npm run typecheck && npm run lint
```

* **类型守卫**：如果是 TypeScript 项目，保持 `tsc --noEmit` 在后台热监听。一旦 AI 制造了隐式 `any` 或类型不匹配，立即报警，并在下一轮对话中把类型报错直接喂给 AI 纠偏。
* **代码格式卫士**：利用 Prettier 与 ESLint，强行把 AI 的小毛病在格式化阶段就消灭干净，确保代码库的血统纯正。

## 另起炉灶的艺术：什么时候该继续对话，什么时候该开启全新 Chat？

这是一门极难掌握、但在实战中至关重要的心智决策艺术。

当对话陷入泥潭（AI 连续 3 次给出的代码都有编译错误，并且它开始反复为同一个 Bug 道歉）时，继续在这个染缸里纠缠是极其愚蠢的。

### 🚨 黄金决策法则
* **继续对话的场景**：
  * 任务尚未完成，但 AI 的思路与架构设计完全在你的轨道上，且当前的微调是顺畅的。
  * 当前会话总长度较短（Token 消耗低）。
* **开启全新 Chat 的场景**：
  * AI 开始出现“失忆”或“幻觉”现象（上下文腐烂）。
  * 发生大面积报错，需要切换思路，采用另一种完全不同的重构设计。
  * 当前阶段性的微任务（例如：编写完 Unit Tests）已经大功告成，你需要干净的场地开启下一个微任务。

> **“挥别过去的错，开启全新的 Chat，是每个 AI 架构师必须拥有的断舍离胸怀。”**
