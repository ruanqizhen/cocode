# 提示词工程

> 准确地表达自己的思想，是一项我们最宝贵的能力。


我们刚开始使用 AI 编程时，都会经历这样一个阶段，直接对 AI 说：“帮我写个登录系统。”

然后 AI 生成了一大堆代码... 运行... 报错... 修改... 继续报错...

有人到这个阶段就会得出结论：“AI 写代码不行。”

但事实上，我们是有办法让 AI 做的更好的。

这就好像是，你对一个刚入职的实习生说：“帮我做个登录系统。”一样，他大概率也不知道你到底想要什么。

他会迷惑，是邮箱登录？还是手机号登录？要不要验证码？密码是否加密？是否支持找回密码？是否需要管理员后台？

需求不明确，AI 就只能瞎猜，结果就会很随机。

而提示词工程（Prompt Engineering）的本质，其实就是一门沟通艺术。它帮助我们把脑海里的想法，准确地传递给 AI。


## 拒绝模糊指令

正如上面的例子，如果给 AI 的指令是模糊的，AI 就只能猜。它猜对了，是运气。猜错了，才是常态。

在工程化场景中，一条高质量指令通常具备三个特征：明确、具体、有约束。

### 明确（Clarity）

首先要告诉 AI 最终目标是什么。不要让 AI 猜。

比如，“我想提升这个函数性能”，就是个模糊表达。因为 AI 根本不知道你是觉得它慢、耗内存、还是网络请求太多。

明确表达应该是这样的：“> 这个函数在处理十万条数据时耗时超过 3 秒。请分析时间复杂度，并通过减少重复遍历和内存分配，将其优化到 O(N)。”

目标越明确，AI 越容易生成接近我们预期的结果。

### 具体（Concreteness）

这是一个非常不具体的要求：“帮我连一下数据库”。AI 并不知道应该用哪种方案链接。

为了清晰描述问题，除了目标，我们还要告诉 AI：

* 输入是什么；
* 输出是什么；
* 使用什么技术；
* 在什么环境运行。

例如：

“ 使用 Supabase JavaScript SDK。 输入用户 ID。 查询 profiles 表。 返回 avatar_url 字段。 使用 TypeScript 编写。”

把这些具体要求交给 AI，它才不容易误入歧途。

### 约束（Constraints）

很多时候，除了告诉 AI 做什么，还需要告诉它不能做什么。

例如：

- “禁止使用第三方工具库。”
- “不允许出现 any 类型。”
- “不要改动原有代码，新的功能另写一套函数”

这些约束看似不起眼，却能极大降低 AI 产生偏差的概率。经验丰富的开发者往往花更多篇幅描述限制条件，而不是功能本身。因为功能大家都能实现。真正困难的是在限制条件下实现功能。


## AI 编程中的黄金公式

提示词太短，很容易就不够明确和具体，但也不是越长越好。

一个优秀的提示词通常由四部分组成：

- 背景（Context）
- 目标（Goal）
- 约束（Constraint）
- 输出格式（Output）


例如：

```text
背景：
我要开发一个单文件 HTML 的待办事项工具。

目标：
支持添加、删除、完成任务。

约束：
不允许使用任何第三方库。
必须使用 LocalStorage 保存数据。
适配手机浏览器。

输出：
直接给出完整 HTML 文件。
```

你会发现：

当信息结构清晰以后，即使提示词不长，AI 的输出质量也会明显提高。



## Context + Instruction 模式

这是软件开发中最常用的提示模式。

简单来说：

先告诉 AI 当前环境是什么。

再告诉 AI 该做什么。

例如：

```markdown
# Context

当前项目使用 Node.js + Express。
我们正在开发一个用户系统。

# Instruction

请编写一个生成 JWT Token 的函数。

# Constraint

Token 有效期为 7 天。
需要处理异常情况。
```

这就像你先带新同事参观工位，再给他安排任务。

如果连项目背景都不说，AI 很容易做出与实际环境不兼容的方案。



## Few-Shot：给 AI 看范例

很多时候，与其解释半天，不如直接给例子。这就是 Few-Shot（少样本学习）的核心思想。

假设你们团队有统一的 API 返回格式：

```json
{
  "success": true,
  "data": {},
  "message": ""
}
```

那么与其长篇大论解释格式规范，不如直接告诉 AI：“ 请严格按照下面示例的风格编写后续接口。” 然后贴出一个标准案例。 AI 往往会瞬间理解你的意图。

对于代码风格、页面设计、文章写作、游戏界面，这种方法都非常有效。很多时候，一个优秀示例的价值，胜过一千字解释。


## Role 模式：给 AI 一个身份

AI 会根据身份设定自动调整回答风格。

例如：

当你分别告诉 AI :“你是一位拥有十年经验的前端架构师”和“你是一位严格的软件安全审计专家”。之后，得到的答案往往完全不同。前者关注工程实现。后者关注漏洞风险。

所以，当我们在做网页游戏，可以这样说：“你是一位资深街机游戏设计师。”；在写个人主页时候可以说：“你是一位顶级 UI/UX 设计师。”；在排查 Bug 时，告诉 AI“你是一位经验丰富的系统调试专家。”

一个好的角色设定，相当于帮 AI 戴上了一副特定的眼镜。



## 深度对话：不要频繁开启新会话

如果 AI 回答得不满意，就立刻开个新的聊天窗口，重新来一遍。这种做法往往会降低效率。因为 AI 会失去之前积累的全部上下文。

更高效的方法是，在当前对话里持续反馈。直接告诉 AI 你是哪里不满意，比如“ 按钮样式不错，但太大了。”或者“功能正常，但手机端布局错位。”再或者“这个方案可行，但性能仍然不够好。” 等等。

这种持续反馈的过程，本质上就是软件开发里的迭代。 AI 会逐渐逼近你脑海中的目标。



## 学会进入“诊断模式”

如果 AI 生成的程序出错了，不要直接问“为什么运行不了？”

因为这对于 AI 来说，是一个很难回答的问题。AI 很可能无法直接看到程序的运行。

更有效的提问方式是，把你看到的全部信息都反馈给 AI，比如：

```text
现象：
点击按钮后页面白屏。

报错：
Uncaught ReferenceError:
userList is not defined

已排查：
接口正常。
数据库正常。

请分析最可能的三个原因，
并告诉我如何验证。
```

这类问题通常能得到质量极高的回答。因为你给了 AI 足够的信息去推理，而不是让它凭空猜测。


## 提示词的终极秘诀

很多文章喜欢把提示词工程描述成一种神秘技术。仿佛掌握几个特殊咒语，就能让 AI 瞬间变强十倍。

事实上并非如此。真正优秀的提示词，并不复杂。它们只是体现了一个简单原则：“把 AI 当成一个聪明、勤奋，但不读心的实习生。”

不要指望它猜测你的意图。不要默认它理解你的背景。不要认为它知道你的偏好。把需求说清楚。把目标讲明白。把约束列完整。把问题描述准确。


提示词工程并不是在学习如何与机器沟通。而是在学习如何更准确地表达自己的思想。而这种能力，无论有没有 AI，都是我们最宝贵的能力之一。


对于一个小程序来说，我们之前所学的方法基本可以应付了。但是对于一个再复杂一点的项目，哪怕只是包含两三个文件的程序，用上面这些方法往往就不太够用了。我们需要用到专门的编程工具来帮助我们开发项目。
