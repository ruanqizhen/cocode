# 谋定而后动：规划与执行的绝对分离

> **“在代码行开始跳动之前，战争就已经在架构图和 Spec 里分出了胜负。”**

---

## 4.1 自动售货机陷阱：为什么直接要代码通常会得到垃圾？

很多开发者对待 AI 的方式就像使用“自动售货机”：投一枚硬币（丢一段模糊的话），就期望掉出一瓶完美的饮料（一段可以直接上线、性能拉满、没有 Bug 的代码）。

这正是导致代码危机和“屎山爆棚”的罪魁祸首——**自动售货机陷阱**。

### 为什么直接要代码必输？
1. **认知跨度的坍塌**：大模型需要在一句话的提示词中，同时完成“理解业务逻辑”、“进行架构设计”、“选择数据结构”、“编写代码”、“编写异常处理”等多维思考。这会导致它的注意力（Attention）极度分散。
2. **缺乏纠偏缓冲区**：AI 一旦直接吐出成百上千行的代码，人类在视觉疲劳下根本无法对每一个细节进行严密审查，最后只能闭着眼睛复制粘贴。

:::important 铁律
**绝对不要让大模型在没有输出规划（Plan）的前提下直接编写非平凡（Non-trivial）的代码。**
:::

---

## 4.2 编写功能规格说明（Spec）：如何用自然语言定义完美的目标与约束

在动工之前，我们需要写一份极其精简但明确的**功能规格说明书（Spec）**。不要被这个词吓到，在敏捷人机协作中，一份 Spec 往往只需要几百字。

### 一个优秀 Spec 的黄金骨架
* **【目标（Goal）】**：用一两句话讲清楚我们要解决什么问题、为谁服务。
* **【用户故事/使用场景（User Scenarios）】**：列出最核心的一到两个高频使用流程。
* **【技术栈与接口契约（Tech Stack & Contracts）】**：指定文件位置、输入参数类型、返回值类型（例如：Prisma Schema, TypeScript Type）。
* **【死命令约束（Critical Rules）】**：
  * “必须处理空值，返回 404 状态码。”
  * “禁止引入外部大依赖，只用原生标准库。”

## 4.3 实施计划（Plan）的诞生与评审：拆解大任务，让人类掌握架构裁判权

有了 Spec 之后，下一步是命令 AI：“**根据这份 Spec，请先不要写代码，为我编写一份详细的实施计划（Implementation Plan）**。”

### 为什么要先出 Plan？
1. **降低思维熵值**：将大任务拆解为清晰的步骤（例如：Step 1 修改 Schema，Step 2 编写 Service，Step 3 补充 Test Case）。
2. **人类的架构卡点**：人类在阅读 Plan 时，可以以极快的速度发现 AI 的设计漏洞（例如：“第 3 步不对，我们不应该在 Service 里直接写 SQL，必须通过 Repository 层”）。
3. **在根源掐死偏差**：纠正一个 Plan 只需要几秒钟的对话；而纠正已经写出来的 1000 行错误代码，可能要花上几小时。

---

## 4.4 实战演练：玩转 Cursor 的 Plan Mode 与偏离轨道时的回退艺术

让我们通过实际场景演练，如何在 Cursor 或 Cline 中将这一套“谋定而后动”的工作流发挥到极致。

### 🛠️ 步骤一：进入 Plan Mode
在 Cursor Composer 或 Cline 中，首先下发明确指令：
> “使用 Plan 模式。我需要为用户购物车新增‘限时打折’功能。请先输出实施计划，标记出需要新建和修改的文件，等待我确认后，才能转入 Act 模式执行。”

### 🔬 步骤二：评审与对齐
AI 吐出 Plan 后，人类冷眼审视：
* “我批准你的第 1、2 步。但是第 3 步中的打折算法，必须单独封装进 `utils/discount.ts` 中，不要写在主业务类里。”
* AI 调整 Plan 并输出最新版。

### 🚀 步骤三：Act 执行与回退（Rollback）
一旦你打出“批准执行（Approved）”，AI 就会行云流水地改写代码。

:::tip 回退的艺术
在执行过程中，如果发现 AI 偏离了最初的架构规划，或者编写的代码产生了离奇的报错，**千万不要在错误的道路上继续打补丁**。

**果断使用 Git Rollback，将代码回退到本步骤动工前的干净状态**，修改你的约束提示词，开启全新的 Chat 会话重新生成。小步快跑，及时止损，才是真正的编程修行。
:::
