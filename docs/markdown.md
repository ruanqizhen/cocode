# Markdown 文件格式

> 代码是写给机器运行的，Markdown 是写给人类和 AI 共同阅读的。

你可能已经发现了，在与 AI 的交互中，我们一直使用的是 Markdown 格式。你可能会好奇：为什么整个 AI 协作生态都在高度默契地拥护 Markdown（.md）格式，而不是更高级的 Word 或者是结构化的 JSON/XML？

答案非常纯粹：Markdown 是目前对抗 LLM“降智”、实现最高 Token 经济学效率的数字通货。

下面将为你彻底拆解 Markdown 的核心语法，并揭秘如何编写出让 AI Agent 工具欲罢不能的“高纯度文档”。


## 为什么 AI 偏爱 Markdown？

对于大语言模型而言，普通的 TXT 文件缺乏结构，极易导致长文本中的逻辑混淆；而 HTML 或 XML 包含大量的闭合标签（如 `<div>...</div>`），会平白无故消耗大量昂贵的 Token，并引入文本噪音。

Markdown 凭借极其克制的“语法糖”，实现了以下三大 AI 友好特性：

- **极致的 Token 消耗比**：仅需一两个字符（如 `#` 或 `-`），就能向模型无缝传递“标题层级”和“列表关系”。
- **天生的注意力聚焦**：前沿模型在预训练阶段吞噬了全网的 GitHub 仓库和技术博客，它们对 Markdown 语法的骨架有着天然的条件反射，能够精准提取主谓宾与核心约束。
- **完美的 Diff 审查基准**：无论是 Cursor 的 Composer 模式还是 Aider 的自动文件修改，Markdown 的纯文本属性让 AI 代理能够以最小的行级计算开销，为你呈递出最直观的代码差异预览。

## 核心语法速查表

以下是编写项目文档与机器规则时高频使用的标准 Markdown 语法：

### 文本层级与骨架

```markdown
# 一级标题（通常用于项目名称或根规则文件）
## 二级标题（用于大模块，如：架构分层、环境配置）
### 三级标题（用于具体子项说明）

```

> 💡 **AI 调优贴士**：强烈建议在文档中保持严谨的标题嵌套。AI 编程工具在做 RAG 检索时，往往会按照标题块进行内容切片（Chunking）。如果你的层级混乱，AI 捞上来的片段就会断章取义。

### 列表与条件约束

```markdown
- **无序列表项 1**：使用星号或减号。
- **无序列表项 2**：适合陈述并列的系统约束。

1. **有序列表项 1**：强制执行的第一步。
2. **有序列表项 2**：必须紧跟的第二步。

- [ ] **待办事项 1**：尚未实现的开发蓝图。
- [x] **已完结事项 2**：已跑通的系统基建。

```

### 代码块与高亮（核心语法）

对于 AI 编程工具而言，代码块的声明格式直接决定了它能否为你做精准的语法高亮与行级 Diff 改写。

```markdown
```typescript
// 必须在反引号后显式指定语言声明（如 typescript, python, json）
export function greet(name: string): string {
    return `Hello, ${name}!`;
}
```

```

:::tip 
千万不要写不带语言后缀的孤立代码块（即单纯的 ``` ）。类似 Claude Code 这样的终端工具，一旦无法识别代码块的 AST 属性，就会放弃局部修改，转而简单粗暴地将几百行的整个文件重新重写一遍，极易引发会话中断。
:::

### 表格与关系映射

表格是为 AI 堆砌“规则对立面”或“数据流向”最高效的编排武器：

```markdown
| 路由路径 | 关联服务 | 权限级别 |
| :--- | :---: | ---: |
| `/api/v1/auth` | AuthService | Public |
| `/api/v1/user` | UserService | User_Role |

```

## 面向 AI 的“上下文级别” Markdown 高级重构

理解了基础语法后，我们要如何对文档进行升维，才能让它变成一套完美的“机器可读基础设施”？

### 策略一：善用 YAML Front Matter（全局前缀常驻）

在大型项目文档（如 `ARCHITECTURE.md`）的最顶部，使用双横线包裹一段键值对，这被称为 Front Matter。现代 AI 代理在读取文件时，会优先将其解析为全局元数据：

```markdown
---
project: "Anachron Forum"
version: "2026.06"
primary_tech_stack: ["React", "TypeScript", "Prisma", "Express"]
strict_constraints: "No external crypto packages allowed"
---

# 架构全局规范
...

```

这能让进入项目的 Agent 瞬间建立“大局观”，在其内部的规划器（Planner）中一秒对齐技术栈版本，防止它用老旧版本的语法污染你的现代代码。

### 策略二：利用“区块引用”制造强力隔离带

当你需要向 AI 警示项目中的“雷区”或“绝对不准触碰的安全底线”时，不要将它们埋在普通的正文段落里。使用 `>` 符号将其转化为高内聚的区块引用：

```markdown
> ⚠️ **底层核心架构防线（Core Boundary）**
> 
> 本工程中所有的数据库查询动作，**必须** 走 `src/lib/prismaClient.ts` 导出的单例对象。
> **绝对禁止** 在任何业务逻辑层或路由层中私自使用 `new PrismaClient()` 重新建立连接，否则会导致高并发下的连接池耗尽死锁！

```

现代 AI 编程工具的 Harness 模块在动态剪裁 Prompt 时，会赋予 Blockquote 更高的权重，在模型底层的注意力机制（Attention Matrix）里，这种格式往往能够获得更高的评分加权。

### 策略三：结构化你的机器规则（`.cursorrules` 最佳实践范本）

不要将给 AI 的规则写成人性化的散文。以下为你提供一个严格符合 Markdown 上下文工程学的规则模版，你可以直接复制并修改为你项目根目录下的 `.cursorrules` 或 `AGENTS.md`：

```markdown
# 行为准则（AI Agent Behavior Protocol）

## 1. 认知语境（Context）
- **目标项目**：高并发历史人物对谈论坛（Echo Hall）
- **开发原则**：坚守强类型契约，追求零运行时错误。

## 2. 动作触发流（Action Loops）
### 编写代码前：
1. 必须使用语义搜索确认项目内是否已存在同类的工具函数。
2. 强制在侧边栏输出 3 行以内的 `Implementation Plan`（执行计划架构），等待人类确认。

### 编写代码后：
1. 自动调起终端运行 `npm run lint` 进行静态检查。
2. 若遇到报错，禁止向人类求助，必须自主读取错误日志并至少尝试自主修复 3 次。

## 3. 拒绝答复边界（Negative Bounds）
> - 绝对禁止使用 `var` 关键字。
> - 绝对禁止绕过全局错误拦截器（`errorHandler.ts`）。
> - 遇到复杂的业务不确定性时，禁止盲猜，立刻停下动作并向人类发出确认澄清。

```
