# 大模型时代的编程心法

> 心法是招式之本。

我们在网上经常会看到各种所谓的“神级提示词”、“万能提示词模板”、“一句话生成整个 App”之类的浮躁宣传。但真正深入将 AI 接入工作流之后，你就会逐渐发现，人与 AI 协作的核心，绝对不是“背提示词”。

不同的问题，不同的语境，最有效的提示词一定是动态变化的。所谓的万能模板，往往流于“万金油”——处处能用，也就处处都不好用。

如果能转换视角，从“系统设计”的角度去理解大语言模型（LLM），看清它到底是如何思考、如何遗忘、如何犯错的，我们就能更高效地驾驭 AI，让它成为真正的生产力杠杆。

---

## LLM 的运行内幕

大型语言模型本质上并不是“电子大脑”，也没有真正意义上的自主意识。它更像一个极其庞大的**概率预测系统**。它所做的一切，都建立在核心的底层逻辑之上：**“根据当前已有的内容，预测下一个最可能出现的 Token。”**

而我们所感受到的“智能感”，其实就是这种海量的概率计算在复杂的语言结构上，所产生的一种“涌现”结果。

### Token（词元）

**Token** 是大模型世界里的最小语言单位。在大模型内部，它无法像人类一样直接按“字”或“单词”来理解语言，而是需要把一切输入拆解成 Token。

* **在英文世界中：** 一个单词通常对应 $1 \sim 1.5$ 个 Token（大约 4 个连续字母组成一个 Token）。
* **在中文世界中：** 情况会稍微复杂一些。早期的模型通常一个汉字就会占用 $1 \sim 2$ 个 Token；而如今针对中文优化过的现代大模型（如国内主流模型及最新的开源模型），常用的词组、成语甚至短句会被聚合为一个 Token。也就是一个汉字不到一个 Token。

### 上下文窗口（Context Window）

**上下文窗口** 本质上就是 AI 当前能“记住”的信息边界，它包含了你发送的代码、历史聊天记录以及 AI 之前的回答。

虽然现在的模型动辄宣传支持 “1M（百万级）Context” 甚至超长上下文，但实际的工程表现并没有想象中完美。在长文本处理中，依然普遍存在著名的 **“Lost in the Middle（迷失在中间）”** 现象：模型对上下文开头和结尾的信息最为敏感，而处于中间的大量内容则容易被注意力机制稀释。

这非常类似于人类阅读长篇财报或技术文档时的状态：第一页和最后一页印象最深，中间几十页看过就忘。

因此，喂给 AI 的信息绝不是越长越好。为了防止核心逻辑被稀释，我们需要主动控制上下文长度，定期重置对话以清除历史包袱。在必要时，还需要利用 `README`、规则文档或架构摘要来“锚定”模型，反复强调核心约束。

### 温度参数（Temperature）

**温度参数（Temperature）** 是控制模型输出创造力的调节阀。它控制的不是冷热，而是**输出结果的随机性**：

* **当 `Temperature = 0` 时：** 模型会极度保守，永远选择概率最高、最稳定的那个 Token。这意味着针对相同的输入，输出结果是确定且可重复的。这种模式最适合用来**编写代码、调试程序、运行数学计算**等严谨、强调确定性的任务。
* **当 `Temperature = 0.9` 时：** 模型会开始“放飞自我”，乐于尝试那些概率较低但富有想象力的 Token 组合。这种模式适合 **UI 创意、文案创作、头脑风暴**等需要发散思维的场景。

不同的温度，决定了 AI 截然不同的“工作性格”。在编程任务中，通常建议将温度锁定在 $0 \sim 0.2$ 之间。

## AI 的能力边界

大语言模型是一个非常典型的“偏科天才”。它在某些领域的表现强得离谱，但在某些看似简单的地方，又脆弱得令人发笑。

### AI 擅长的“模式匹配”

在训练阶段，LLM 吞噬了海量的互联网语料，包括无数的开源代码、技术文档和标准教程。因此，它对“成熟模式（Pattern）”有着极其恐怖的复现能力。

诸如实现一个小游戏、写一个数据转换脚本、构建一个基础的网页登录系统等，这些在互联网上被人类重复实现过无数次的常规任务，AI 处理起来**得心应手**。你往往只需一句话，它就能在几秒钟内生成几百行结构完整的代码。但这并非因为它真正“理解”了你的业务系统，而是因为它在庞大的概率海洋里，瞬间匹配到了最接近的既有模式。

### AI 不擅长的“精确推理”

因为 LLM 是**概率机**而非**逻辑机**，它是在“预测”一个看起来最像正确答案的文本，而不是在严密地推导因果关系。

当面对需要严密逻辑闭环的任务时——例如推导复杂的数学公式、设计深层递归逻辑、构建复杂的状态机，AI 的脆弱性就会迅速暴露。更致命的是，AI 从来不会主动承认自己“算不出来”，而是会通过偷换概念、强行续写来编造事实，一本正经地胡说八道。

:::tip 核心策略
真正高效的人机协作模式应当是：**人类负责定方向、设约束、当裁判；AI 负责搬砖、写样板代码、干体力活。**
在团队中，你可以把当前的 AI 当成一个执行力极强、手速极快的初级工程师（Junior Developer），但千万不要盲目把它当成完全可信的系统架构师。
:::

## 幻觉剖析

**幻觉（Hallucination）** 是大语言模型在编程领域最常被诟病的问题。你可能经历过，AI 会极其诚恳地向你推荐一个根本不存在的 npm 包，或者言之凿凿地虚构某个框架的官方 API。

例如下面这段典型的“翻车”代码：

```javascript
// 经典 AI 幻觉：一本正经地调用不存在的 API
const response = await aiModel.nonExistentSuperAwesomeFeature({
  ultraTurboMode: true
});

```

这段代码命名规范，语法优雅，看起来极其专业。然而，现实世界中这个 API 根本不存在。

### 为什么会产生幻觉？

因为大模型并不是在“查字典”或“检索数据库”。如前文所述，它是在做**统计学意义上的语言续写**。

如果它在你的上下文提示中，发现某种写法“非常符合某种优秀的、主流的 API 设计风格”，它就会顺着这个概率趋势把代码补全出来，哪怕这个 API 纯属虚构。为了保证文本流的连贯性，它宁可硬着头皮往下编，也不会轻易抛出“我不知道”的异常。**幻觉，是概率模型与生俱来的副作用。**

### 如何对抗幻觉？

只要我们还在使用基于概率预测的 Transformer 架构模型，就无法从根本上完美“消灭幻觉”。在工程实践中，对抗幻觉的唯一有效路径是**建立“验证体系”**。

这要求我们在接收到 AI 提供的方案后，必须通过**运行单元测试、查阅官方标准文档、静态代码检查**等手段去交叉验证。这也正是为什么在大模型时代，一个优秀程序员的核心竞争力不再是“让 AI 一次性生成完美代码”，而是**能敏锐地一眼看穿 AI 在哪里胡说八道，并具备快速纠错的能力**。

## 思维范式的跃迁

在传统编程时代，一位优秀程序员的护城河往往建立在记忆力与熟练度上：熟记上千个 API 接口、精通各种框架的语法细节。那时候，代码的“肌肉记忆”本身就是核心生产力。

然而，AI 的崛起正在加速重塑这个行业的底层逻辑。在记忆容量和检索速度上，人类根本无法与模型匹敌，“背诵 API”的技能正在迅速贬值。未来的价值链发生了重组：**“怎么写”变得越来越廉价，“写什么”和“为什么这么写”变得越来越值钱。**

* **过去：** 程序员更像是一个手艺人、工匠，专注于一砖一瓦的雕琢。
* **未来：** 程序员则更像是一个设计师、导演。

编程的核心工作，正在从“逐行手写代码”悄然跃迁为“组织核心逻辑、调度通用智能、管理上下文环境”。这是一种深远而深刻的思维范式变革。

## 上下文控制权

当你在实际项目中高强度使用 AI 编程几个月后，必然会得出一个深刻的结论：**单纯研究孤立的提示词是远远不够的，真正分出高下的是“上下文控制（Context Control）”。**

**谁掌控了上下文，谁就掌控了 AI。** 因为模型接下来的所有思考和输出，都完全戴着当前上下文剪裁出来的“思想链条”。

在软件工程中，以下要素共同构成了一个完整的项目上下文：

* **README / 项目宣言：** 明确项目的核心定义与技术栈。
* **项目规则（Rules）：** 约束架构边界与设计哲学。
* **文件结构（Structure）：** 规范模块划分与依赖关系。
* **命名规范（Naming Convention）：** 保持代码风格的高度一致。
* **状态摘要（Status Summary）：** 记录当前开发进度与阻碍。
* **TODO 列表：** 明确下一步的精确动作。

我们投喂给 AI 的，不应该只是一个“帮我写个函数”的孤立命令，而应当是一个**结构清晰、可以被持续维护和动态更新的“认知环境”**。管理好这个环境，AI 才能持续稳定地输出符合预期的最高质量代码。
