# 上下文工程（Context Engineering）：喂饱你的 AI

> **“大模型没有记忆，你提供给它的上下文，就是它灵魂的全部容器。”**

---

## “上下文即一切”：为什么 90% 的 AI 错误都源于信息贫瘠？

很多开发者在遇到 AI 报错或写出低级代码时，第一反应是“这个模型太笨了”。但如果冷静地做一次“死后剖析（Post-mortem）”，你会惊讶地发现：

> **90% 的 AI 编程失败，根本原因都在于输入给它的上下文极度匮乏、模糊或充满噪音。**

如果大模型不知道你当前的数据库版本、不清楚你全局的错误处理类、看不到你定义的公共 TypeScript 接口，它就只能依据公开互联网的“通用概率平均值”去盲猜。盲猜的结果，自然就是与你的系统格格不入的垃圾代码。

---

## 长期上下文构建：编写高效的 README 与架构图景文档

为了让 AI 随时保持清醒，我们必须在代码库中埋下**长期上下文（Long-term Context）** 的锚点。

### 📁 项目说明书（README.md）
不要只写一句“这是一个 React 项目”。一个对 AI 极其友好的 README 应当清晰列出：
* **技术栈明细**：具体到 Next.js 14 (App Router), Prisma, TailwindCSS, PostgreSQL 等。
* **目录架构设计**：简述 `/components`、`/lib`、`/services` 各自的职责。
* **核心规约**：如“所有数据请求必须通过 Server Actions 进行”。

### 🎨 架构图景与领域模型（ARCHITECTURE.md）
用 Markdown 详细记录项目的全局业务实体模型和设计思想（例如：什么是 `Tenant`，什么是 `Subscription`，它们在数据库中的映射逻辑是什么）。当 AI 读懂了架构蓝图，它吐出的每一行代码都会精准归位。

## 即时上下文策略：精准使用 @ 符号，拒绝全量代码库“投喂”

随着大模型上下文窗口越来越大，很多人喜欢直接把整个项目库“投喂”给 AI。这不仅会导致昂贵的 Token 浪费，还会引入巨大的信息噪音。

:::important 精准喂食原则
**只喂它需要呼吸的空气，不要把它淹没在无关的信息海洋中。**
:::

在 Cursor 或 Cline 中，灵活使用 `@` 符号，实施高超的**即时上下文管理**：
* **`@Files`**：只 @ 当前需要修改的文件，以及它直接 import 的依赖文件。
* **`@Codebase`**：只在需要全局重构、寻找特定 API 签名或跨文件引用分析时，才启用 Codebase 级别的语义检索。
* **`@Web`**：当且仅当需要查询最新发布的第三方库文档或解决前沿报错时，才开启实时联网搜索。

## 警惕“上下文腐烂”（Context Rot）：应对长会话崩塌的断舍离技巧

你是否经历过：在同一个 Chat 会话里和 AI 聊了 2 个小时，起初它智商拉满，但随着时间推移，它开始颠三倒四，甚至忘记了 20 分钟前你们刚刚商量好的架构？

这就是经典的**上下文腐烂（Context Rot）**。

### 为什么会发生上下文腐烂？
1. **注意力注意力溢出**：随着历史对话越来越长，核心指令和上下文在长序列中被“淹没”或注意力权重下降。
2. **错误记忆累积**：会话中曾经讨论过、但被否决的错误方案依然残留在历史记录中，干扰了 AI 的判断。

### 🌟 断舍离技巧：另起炉灶的艺术
一旦发现大模型出现智商退化的迹象，**立即关闭当前会话**，并开启一个全新的干净 Chat！
* 在新会话的开头，直接投喂当前最干净的代码，以及一句总结：
  > “我们正在开发用户购物车模块。刚才已经完成了 Schema 变更，以下是目前最新的 `cart.ts` 代码。现在，让我们开启下一步：编写购物车结算的 API。”

## 提效百宝箱：巧用 gitingest 与 repo2txt 打包源码上下文

有时我们需要在网页端大模型（如 ChatGPT Plus 或 Claude CLI 之外）分析整个项目，或者在初始化一个新的 Agent 时快速帮它建立对项目全局的认识。

这时，你可以使用以下开源利器进行源码打包：

* **Gitingest (gitingest.com)**：一个极棒的工具，输入任何 GitHub 仓库链接，它能瞬间帮你将整个仓库的内容压缩、打包为一个非常整洁的、对 AI 友好的单个文本文件（.txt），并自动附带目录树结构。
* **repo2txt / code2txt**：在本地运行的脚本，能够过滤掉 `.git`、`node_modules`、`dist` 等无关目录，将项目的有效源文件拼装为一个 txt 文件，极大地方便了跨工具投喂。

```bash
# 典型的 gitingest 终端使用示意
gitingest /path/to/my-codebase --exclude "*.log,node_modules,dist"
```
掌握这些工具，你就能随时随地为你的赛博伙伴搭建起一座高纯度的知识灯塔。
