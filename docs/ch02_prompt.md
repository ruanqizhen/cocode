# 第二章 提示词工程（Prompt Engineering）的工程化落地

> **“垃圾输入，垃圾输出（Garbage in, Garbage out）。好的提示词是把你的工程理性，翻译成 AI 能够精确执行的电信号。”**

---

## 2.1 拒绝模糊指令：明确、具体与约束的三大黄金原则

许多人抱怨 AI 生成的代码是“屎山”，但他们自己的提示词往往是：“帮我写个登录接口”。这种模糊的指令是烂代码的催化剂。

在工程化场景中，一条合格的指令必须遵循三大黄金原则：

### 🎯 明确（Clarity）
清楚地陈述目标，不留一丝歧义。
* **反面**：“我想提升这个函数的性能。”
* **正面**：“这个函数目前在大数据量下执行缓慢。请分析时间复杂度，并通过避免重复数组分配，将复杂度优化至 O(N)。”

### 🛠️ 具体（Concreteness）
指定输入、输出、使用的数据结构和依赖项。
* **反面**：“帮我连一下 Supabase 数据库。”
* **正面**：“使用 @supabase/supabase-js 库，编写一个 TypeScript 函数，输入用户 ID，查询 `profiles` 表并返回 `avatar_url`。”

### 🔒 约束（Constraints）
为 AI 框定严厉的禁区。
* “禁止使用第三方 Lodash 库，必须使用 Vanilla JS。”
* “如果输入为空，必须抛出 `ValidationError` 且附带 400 状态码。”
* “代码中不允许出现任何 `any` 类型。”

## 2.2 核心提示模式演练：Context + Instruction 模式、Few-Shot 示例模仿、角色扮演模式

我们并不需要死记硬背成百上千个提示词模板，只需要熟练掌握以下三个在软件工程中最核心的提示模式：

### 1. Context + Instruction 模式（上下文 + 指令）
这是最经典的结构化提示词。将背景环境与执行命令清晰剥离。

```markdown
# Context
当前项目是一个使用 Node.js + Express + Prisma 构建的 Monorepo 后端服务。
我们正在重构用户鉴权模块，以支持多设备登录。

# Instruction
请为 `auth.service.ts` 文件编写一个生成 JWT 刷新 Token（Refresh Token）的方法。

# Constraint
1. Token 必须存入 Redis，有效期为 7 天。
2. 必须处理 Redis 连接失败的边界异常。
```

### 2. Few-Shot 示例模仿（少样本学习）
当我们需要 AI 输出极具个性化、符合特定内部规范的代码时，Few-Shot 是最强杀手锏。先给 AI 一到两个完美的示例，它会顺着这股逻辑流精准输出。

:::tip Few-Shot 模版
“这是我们项目的 API 响应规范，请模仿它编写接下来的接口：”
*(这里附带 1 个标准的输入与输出代码)*
“现在，请为订单创建接口编写同样格式的代码：”
:::

### 3. Role-Playing 模式（角色扮演）
让 AI 扮演特定的专家，可以瞬间调整它的“上下文先验知识库”，吐出极具专业度的回答。
* “你现在是一位冷酷无情的资深安全审计专家，请审查以下 Rust 智能合约，找出所有可能导致重入攻击（Reentrancy Attack）的安全漏洞。”

## 2.3 深度对话技巧：如何通过连续追问纠偏输出，利用“诊断模式”让 AI 寻找病灶

一次对话就想拿到完美代码是不切实际的。高水平的程序员擅长通过“人机对话流”对 AI 进行动态微调。

### 1. 连续追问与反馈闭环
当 AI 输出不完美时，不要放弃该会话，更不要直接把垃圾代码贴回项目。在当前会话中指出它的具体不足：
> “你刚才提供的代码没有处理并发请求时的锁机制。如果在高并发下可能会产生脏读。请加入 Redis 分布式锁，并重新生成这部分逻辑。”

### 2. 巧用“诊断模式（Diagnostic Mode）”
不要直接问“为什么我这段代码跑不通？”，改用系统化的诊断视角逼迫 AI 帮我们排查：

:::info 诊断模式 Prompt 结构
1. **输入当前现象**：抛出的 Stack Trace、完整日志和异常退出码。
2. **输入已排除的嫌疑**：我已经检查过数据库连接、配置文件均正常。
3. **指令**：请扮演一位系统底层专家，列出导致此现象的 top 3 潜在原因，并为每个原因设计一条最小复现（Reproduction）命令。
:::
通过这种极具工程感的方法，你将发现大模型会以惊人的效率锁死真正的 Bug 病灶。
