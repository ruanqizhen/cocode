# 上下文工程

> 大模型没有记忆，你提供给它的上下文，就是它灵魂的全部。


在之前的尝试中，你是否曾花费半个小时精心打磨一段长长的提示词（Prompt），期望 AI 助手为你生成一段完美无瑕的代码呢？是否成功了呢？

如果你的项目是一个比较复杂的项目，那么你的尝试多半会失败。这并不完全是因为模型太笨了，90% 的 AI 编程失败，根本原因都在于输入给它的上下文极度匮乏、模糊或充满噪音。如果你在提示词中不提及项目的背景，大模型就只能依据公开互联网的“通用概率平均值”去盲猜。

目前，业内的一个共识是：决定 AI 编程能否生成生产级代码的关键，不再是遣词造句的提示词工程（Prompt Engineering），而是对动态信息流精细调配的上下文工程（Context Engineering）。

## 什么是上下文工程？

著名 AI 研究员 Andrej Karpathy 曾为上下文工程给出了一个被广泛引用的经典定义：

> “上下文工程是一门精妙的艺术和科学，它旨在为下一步操作向上下文窗口中填充恰到好处的信息。” —— Andrej Karpathy

如果把大型语言模型（LLM）比作高速运转的 CPU，那么它的上下文窗口（Context Window）就是其 RAM（内存）。正如操作系统必须精心管理 RAM 中加载的活动进程与数据一样，开发者也必须精心策划 AI 在生成当前行代码时所能看到的所有周边信息。在软件开发语境下，上下文工程是设计和构建 AI 代理（Agent）整个动态信息生态系统的学科。

### 上下文工程 vs. 提示词工程

为了厘清两者的本质区别，我们可以通过以下多维度矩阵进行对比：

| 维度 | 提示词工程 (Prompt Engineering) | 上下文工程 (Context Engineering) |
| --- | --- | --- |
| **核心关注点** | 如何遣词造句，侧重于指令的表达文本与语气 | 模型在当下知道什么，侧重于知识包的供给与纯度 |
| **操作性质** | 静态的、一次性的任务描述与模板定义 | 动态的、交互迭代式的信息系统网格管理 |
| **解决的痛点** | 意图表达不清，生成方向偏离预期 | 代码幻觉、脱离项目实际的“空中楼阁”代码 |
| **适用场景** | 单功能函数生成、通用算法问答、文本格式化 | 复杂项目重构、跨多文件业务编排、Agent 持续循环 |

简而言之：提示词工程是告诉 AI “做什么”，而上下文工程是负责为 AI 提供“安全、精准、高效完成任务所需的全部武器和弹药”。



## 深入剖析现代 AI IDE 的上下文抓取机制

在现代 AI IDE（如 Cursor、Windsurf）或底层的 Agent 框架中，右侧聊天窗口或控制台中的 `@` 符号系统是我们最强大的“喂食铲”。理解它们各自的底层抓取机制，才能做到精准投喂，而不是盲目注入：

* **`@Files`（精准靶向，纯度最高）**：直接读取指定文件的全部文本，以最高优先级注入 Prompt 顶部。Token 最省，相关性最高，是高频重构的首选。它能实现 100% 的无偏见感知。当你需要让 AI 调用某个特定公共接口或修改单文件业务时，直接 `@` 该文件能省去大量的无用检索 Token。
* **`@Codebase`（向量检索，跨文件感知）**：通过本地向量嵌入（Embeddings）进行 RAG 检索，自动匹配并提取与当前问题最相关的 3~5 个代码片段注入窗口。适合在寻找全局未知 Bug、分析跨模块调用流或大型系统重构时使用。它依靠语义距离计算寻找关联代码，但要注意可能存在由于检索泛化带来的噪声。
* **`@Web`（实时联网，打破时间壁垒）**：调起搜索引擎抓取第三方最新文档（如 Next.js 15），转化为 Markdown 格式注入，彻底消灭大模型的离线时效偏差。大模型的训练数据终究是有截止日期的。当引入全新的库、发生重大的主版本 Breaking Changes 或调用外部不稳定的第三方 Webhook 时，必须配合 `@Web` 强制拉取实时规范，防止 AI 凭借过时记忆胡说八道。
* **`@Git`（变更追踪，增量感知）**：自动读取 `git diff` 或近期 Commit 记录，提炼当前开发增量上下文，让 AI 秒懂你最近半小时的修改意图。



## 3. 经典实战对比：Poor Context vs. Rich Context

为了切身感受到上下文管理对生成质量的决定性影响，我们现场模拟一个真实后端开发任务：**“在 Express 框架中编写一个用户注册（Register）路由函数”**。

### ❌ 场景 A：极度贫瘠的上下文 (Poor Context)

**开发者在对话框中直接草率输入：**

> “帮我写一个 Express 的用户注册接口函数。要能保存用户，密码要加密。”

AI 在完全不知道项目任何架构细节的“信息荒漠”中，只能凭借互联网公开语料的“概率平均值”盲猜写出以下代码：

```javascript
// AI 盲猜生成的泛化代码
const express = require('express');
const router = express.Router();
const bcrypt = require('bcrypt');
const User = require('./models/User'); // ❌ 盲猜的文件路径

router.post('/register', async (req, res) => {
  try {
    const { email, password } = req.body;
    // ❌ 盲猜的加密算法，可能与项目全局已有工具类冲突
    const hashedPassword = await bcrypt.hash(password, 10); 
    const newUser = new User({ email, password: hashedPassword });
    await newUser.save();
    return res.status(201).json({ message: 'User created' });
  } catch (error) {
    // ❌ 粗暴的局部脏捕获，直接绕过了项目统一的全局错误处理器
    return res.status(500).json({ error: error.message }); 
  }
});

```

#### 致命缺陷分析：

1. **数据库断层**：AI 盲猜你用的是 Mongoose(MongoDB)，但你的项目工程实际使用的是 Prisma 配合 PostgreSQL。这段代码丢进项目直接导致编译挂掉。
2. **逻辑重复与安全隐患**：密码哈希它引入了新的 `bcrypt` 包，而你项目团队内部明明在 `src/utils/crypto.ts` 中统一封装了更安全的 `scrypt` 加密算法。
3. **架构污染**：它写了简陋的 `try-catch` 并直接向客户端返回原始错误，彻底绕过了你精心编写的全局统一异常拦截器，导致服务监控系统（如 Sentry）无法捕捉到该异常日志。

---

### 🟢 场景 B：精准富集的上下文 (Rich Context)

**开发者通过精准的 `@` 系统，为 AI 搭建了一个极高纯度的认知沙盒：**

> “请帮我编写用户注册的 API 接口函数。我已把核心的依赖关系附在下方：
> * 数据库模型定义：`@schema.prisma`
> * 输入校验契约：`@authValidation.ts`
> * 全局密码加密工具：`@crypto.ts`
> * 全局错误处理器：`@errorHandler.ts`
> 
> 
> 请严格复用这些现有模块，不要引入任何新的第三方加密包，确保通过错误处理中间件安全抛出异常。”

AI 在知识灯塔的指引下，写出的代码完全融入了现有的系统生态：

```typescript
import { Request, Response, NextFunction } from 'express';
import { prisma } from '../lib/prismaClient'; // ✅ 精准复用项目现有的 Prisma Client
import { validateRegisterInput } from '../utils/authValidation'; // ✅ 严格遵守项目的输入校验契约
import { hashPassword } from '../utils/crypto'; // ✅ 精准调用项目的专属加密算法
import { AppError } from '../middlewares/errorHandler'; // ✅ 融入全局统一错误处理器

export async function registerUser(req: Request, res: Response, next: NextFunction) {
  try {
    // 1. 严格使用全局校验机制
    const { error, value } = validateRegisterInput(req.body);
    if (error) {
      throw new AppError(400, `输入校验失败: ${error.message}`);
    }

    const { email, password, nickname } = value;

    // 2. 检查用户是否已存在
    const existingUser = await prisma.user.findUnique({ where: { email } });
    if (existingUser) {
      throw new AppError(409, '该邮箱已被注册');
    }

    // 3. 复用项目中统一的哈希函数
    const hashedPassword = await hashPassword(password);

    // 4. 精准持久化写入
    const newUser = await prisma.user.create({
      data: {
        email,
        passwordHash: hashedPassword,
        nickname
      },
      select: {
        id: true,
        email: true,
        nickname: true,
        createdAt: true
      }
    });

    return res.status(201).json({
      success: true,
      data: newUser
    });
  } catch (error) {
    // ✅ 绝不在局部做脏捕获，安全抛给全局中间件
    next(error);
  }
}

```

#### 完美生成效果：

这段代码达到了“零修改、直接上线”的极高工业级标准。它没有引入任何多余依赖，完美继承了整个工程的编程契约与设计哲学。这就是上下文工程展现出的终极威力。

## 4. 规模化对抗：长会话崩塌与“上下文腐烂”

在人机协作开发中，还有一个悄无声息的隐性杀手叫做“上下文腐烂”（Context Rot）。当你在同一个 Chat 窗口内与 AI 连续对话超过一个小时，积攒了数十上百轮的历史消息后，你会发现 AI 开始变得丢三落四、反应迟钝，甚至连上一轮刚讨论完的函数名都记错了。

### 为什么超长的上下文会导致模型“降智”？

1. **迷失在中间（Lost in the Middle）**：研究表明，即使模型声称支持 100 万的超长 Token 窗口，当实际填充量增大时，其对中部（Middle）信息的提取召回率会呈现指数级下降。你的核心指令很容易被淹没在漫长的历史废话中 [2]。
2. **负面记忆污染**：在长会话调试过程中，你和 AI 一起尝试过、但最终被抛弃的错误改动思路和排错报错，依然残留在历史消息中。这些错误的信息流会持续作为负面样本，严重干扰 AI 当前的推理路径。

### 上下文调配的五大核心策略

为了对抗这种膨胀带来的效率损耗，先进的 AI 代理框架（如 LangGraph、Manus 代理核心）在底层都广泛应用了以下五种上下文调度策略。

1. **写入 (Write) — 外部化记忆**：长周期任务中，允许 AI 将当前阶段的计划、状态或中间结果写入临时文件（如 `scratchpad.md` 或 `todo.md`）。在实践中发现，通过不断重写和更新待办事项列表，能将全局宏观计划持续推入模型的近期注意力范围内，有效避免目标偏离。
2. **选择 (Select) — 精准稀疏化检索**：与其把 100 个相关的源文件都一股脑塞进上下文，不如只提供 5 个最核心的修改目标文件，外加另外 15 个关联文件的浅显“函数签名（Signatures）”或 AST 节点。
3. **压缩 (Compress) — 轨迹自动剪枝**：诸如 Claude Code 这样的前沿工具，会在上下文接近水位极限（如 95% 容量）时自动触发内部压缩机制（Auto-compact）。它调用一个后台模型对先前的历史对话进行核心意图摘要，丢弃逐字逐句的对话流，仅保留已被确认的架构决策和关键变动。
4. **隔离 (Isolate) — 关注点分离（SoC）**：杜绝使用单一长会话 Agent 包揽万物。通过多智能体架构（Multi-agent），将长任务拆分给拥有完全隔离上下文窗口的子 Agent。例如让“文档搜索 Agent”与“测试编写 Agent”各司其职，有效降低单模型的认知负荷与 Token 开销。
5. **排序 (Ordering) — 契合注意力曲率**：由于大模型对最开头和最末尾的信息最为敏感，业内总结出了最优的上下文布局模板，按照敏感度由高到低进行金字塔式排序：
* **【最顶部 · 黄金位置】** 关键核心约束与系统根规则（如：绝对禁止做的事、底层安全契约）。
* **【前半段 · 高召唤区】** 当前可用的核心工具（Tools）声明、核心 API 文档规范。
* **【中部区域 · 缓冲区】** 一般性上下文（项目相关的长历史轨迹、非直接依赖的代码参考示例）。
* **【后半段 · 焦点区】** 动态运行时状态（上一次执行的测试运行输出、Linter 报错信息、终端捕获）。
* **【最尾部 · 强指令区】** 当前最即时的任务目标（开发者正在高亮编辑的文件、用户的当前直接提问）。



### 🛠️ 黄金法则：另起炉灶的艺术

在日常高频开发中，对抗上下文腐烂最朴素、最高效的习惯就是“不要在一个 Chat 会话里养老”。一旦发现 AI 开始出现理解偏差或反应变慢的智商退化苗头，**毫不犹豫地关闭当前 Chat 窗口，开启一个全新的、干净的会话！**

在新会话的开头，直接以最利落的结构“投喂”当前的切片状态：

> “我们正在开发用户模块。刚才已经顺利完成了 `register.ts` 文件的编写，以下是目前最新的完整代码：
> `[在此处粘贴最新的 register.ts 代码]`
> 现在，让我们开启下一阶段任务：编写对应的 `loginUser` 登录认证接口。”

轻装上阵、消除了历史废话干扰的 AI 会在瞬间重新激发其最高级别的代码推理能力。

## 5. 落地指南：构建适合 AI 阅读的上下文基础设施

作为新时代的软件工程师，你的核心工作正从“自己手写每一行代码”演变为“为 AI 编排完美的生产环境”。你不需要去写底层的 RAG 检索算法，但你必须在你的项目工程中建立一套**对 AI 友好的上下文基础设施** [6]：

### 第一步：建立项目级“全景透视图”

不要指望 AI 能够瞬间靠几行代码读懂你的宏观设计。在项目根目录下，维护以下三个标准的、面向机器阅读的 Markdown 文档：

* `PRODUCT.md`：核心业务愿景、主要用户链路与功能边界。（让 AI 知道我们在做什么，避免写出脱离业务逻辑的功能代码）。
* `ARCHITECTURE.md`：项目的系统分层、技术栈选型版本、数据流向设计与核心设计模式。（让 AI 知道我们怎么做，保持架构的一致性）。
* `CONTRIBUTING.md`：日常开发环境配置、Linting 规范、分支管理以及测试用例的编写标准。

### 第二步：编写高可执行力的机器规则（AGENTS.md / .cursorrules）

无论是使用 Cursor 的 `.cursorrules` 还是通用 Agent 的 `AGENTS.md`，不要写成含糊、带有感性色彩的人类散文。规则必须是**具体的、条件触发的、可验证的操作指令**：

| ❌ 糟糕的规则声明（不可执行） | 🟢 优秀的规则声明（精准可执行） |
| --- | --- |
| “请确保你写的代码质量很高，且编写了充分的测试。” | “每次修改业务逻辑文件后，必须立刻自动运行 `npm run test`。当且仅当终端返回退出码为 0 时，才允许向用户汇报任务完成。” |
| “注意不要破坏全局的错误捕获逻辑，多用我们已有的工具。” | “当需要处理接口报错时，禁止在当前函数中使用局部 `try-catch` 块拦截。必须显式抛出 `AppError`，或通过 `next(error)` 传递给全局错误拦截器。” |

### 第三步：面向 KV 缓存友好的 Prompt 设计

如果你正在基于大模型 API 构建团队专属的智能 Agent 系统，务必注意**键值缓存（KV Cache）的优化**。保持系统提示词（System Prompt）前缀的绝对稳定。切忌在 System Prompt 开头注入诸如动态当前时间戳、随机 Session ID 等高频变动变量。这会导致大模型底层的 KV 缓存机制完全失效，导致你的首字生成时间（TTFT）与长上下文推理的 Token 成本飙升 10 倍以上。
